# Tamper Detection Layer — Design

**Date:** 2026-07-23
**Status:** Draft, pending user review
**Owner:** Sharan (Capital One Small Business team)

## Context

Capital One's small business account onboarding flow includes a BKYC step in which
customers upload supporting documents — utility invoices, sales tax permits, EIN
letters, and similar. A prior POC pipeline runs OCR on these uploads, extracts
fields, and matches them against the application.

That pipeline treats the document contents as truthful. It has no way to tell a
genuine utility bill from one that was edited in Acrobat to change the address,
spliced together in Photoshop to change the amount, or generated by an image
model. This spec designs a forensics layer that sits alongside the existing OCR
pipeline and returns a per-document tamper-risk report.

## Goals

1. Detect the four major fraud modes on PDF uploads:
   - **Forgery** (bytes edited after issuance)
   - **AI generation** (whole document or embedded imagery synthesized)
   - **Editing** (fields replaced, text/image overlays)
   - **General inauthenticity** (issuer-template mismatches)
2. Produce an **explainable** report — overall score, decision, and per-signal
   evidence — that an analyst or downstream compliance rule can act on.
3. Ship as a Python library + CLI + FastAPI service so it bolts onto any
   OCR pipeline via import or HTTP.
4. Ship with 10 labeled test documents and an automated evaluation harness.

## Non-Goals

- No persistent storage of documents or reports. Analyze in-memory, return, discard.
- No auth on the FastAPI service. This is a prototype endpoint.
- No integration with the existing OCR pipeline code. The forensics layer accepts
  a PDF (and optionally OCR text the pipeline already extracted) and returns a
  report. Wiring is a separate task.
- No real-world labeled corpus, no ML training, no active-learning loop.
- No PII / data-retention story beyond "in-memory only." Compliance for
  production traffic is out of scope for the POC.
- Per-page runtime target of ~2s is a soft goal, not an SLO.

## Inputs

- **File format:** PDF only — both native (digitally created, e.g. utility
  portal download) and scanned/image-wrapped (phone photo or scan inside a PDF
  container).
- **Optional:** OCR text already extracted by the upstream pipeline, provided
  alongside the PDF. Used by the `ocr_render_diff` detector as a reference for
  what the document *claims* to say.

## Architecture

```

    ▼

    │            text layer, metadata, xref, /Info dict
    ▼

    ▼

```

## Detectors

Each detector implements a common protocol and emits `Finding` objects:

```python
class Finding(BaseModel):
    signal: str
    tier: Literal["structural", "pixel", "ocr_diff", "experimental"]
    score: float                # normalized 0..1 (0 = clean, 1 = strong tamper)
    evidence: dict              # structured, human-inspectable
```

Every detector must return quickly (~2s/page soft target) and degrade
gracefully — if a page has no embedded images, `pixel_forensics` returns an
empty finding list, not an error.

### Tier 1 — `pdf_structural` (deterministic, high-precision)

| Signal | What it catches |
|---|---|
| `producer_creator_mismatch` | Creator says "Microsoft Word" but Producer says "iLovePDF" → re-saved through an editor |
| `mod_date_after_create_date` | /ModDate later than /CreationDate on docs that shouldn't have been edited |
| `incremental_update_present` | Multiple `%%EOF` markers / xref update chain → saved with incremental updates (common Acrobat edit signature) |
| `xref_anomaly` | Malformed / hand-edited xref tables |
| `font_not_embedded` | Non-embedded font on a doc type that normally embeds (utility bill, IRS letter) |
| `font_family_mix_in_line` | Different font families or weights within a single logical line → strong text-replacement signal |
| `text_over_image_overlay` | Text-layer glyphs positioned exactly over a raster region → erase-and-rewrite edit |

### Tier 2 — `pixel_forensics` (scanned PDFs, embedded raster pages)

| Signal | What it catches |
|---|---|
| `ela_hotspots` | Error Level Analysis: re-save at fixed JPEG quality, diff residual; hot regions are suspect |
| `jpeg_ghost` | Region JPEG-compressed at a different quality than the surrounding page → spliced in |
| `noise_residual_inconsistency` | Per-block sensor-noise variance breaks at splice boundaries |
| `copy_move` | Block-matching for duplicated regions (duplicated stamps, signatures, logos) |

### Tier 3 — `ocr_render_diff`

| Signal | What it catches |
|---|---|
| `text_layer_mismatches_render` | Re-render pages at high DPI, re-OCR, diff against supplied OCR text. Large edit distance in specific regions → text layer lies about what's visible |

### Tier 4 — experimental, low-weighted

| Signal | What it catches |
|---|---|
| `ai_generated_image` | Small pretrained SD/GAN detector on each embedded image. Weighted low. |
| `ai_generated_text` | GLTR-style perplexity + burstiness on extracted text. Noisy on formulaic docs → weighted very low. |
| `template_mismatch` | For a starter library of known issuer templates (IRS EIN letter, one utility, one state sales-tax permit): layout hash + logo perceptual hash + expected-field regex presence. |

## Scoring (Approach A — Rule-Based Weighted Scorer)

Each finding contributes `tier_weight × signal_weight × normalized_score` to a
running total. The overall score is `min(1.0, sum_of_contributions)` — a hard
clamp keeps the surface simple and interpretable. Decisions come from the
threshold table.

Weights are hot-reloaded per request in dev so you can tune without a restart.

```yaml
# weights.yaml (starting point — tune against test corpus)
tiers:
  structural: 1.0
  pixel: 0.8
  ocr_diff: 0.9
  experimental: 0.2

signals:
  producer_creator_mismatch: 0.7
  mod_date_after_create_date: 0.6
  incremental_update_present: 0.8
  xref_anomaly: 0.9
  font_not_embedded: 0.4
  font_family_mix_in_line: 0.9
  text_over_image_overlay: 0.95

  ela_hotspots: 0.6
  jpeg_ghost: 0.85
  noise_residual_inconsistency: 0.5
  copy_move: 0.9

  text_layer_mismatches_render: 0.9

  ai_generated_image: 0.4
  ai_generated_text: 0.2
  template_mismatch: 0.6

decision_thresholds:
  pass:   [0.00, 0.30]
  review: [0.30, 0.65]
  fail:   [0.65, 1.00]
```

## Narrative Generator (Approach C — Reporting Only)

After the deterministic report is built, `narrative.py` sends the `findings`
array to Claude (`anthropic/claude-sonnet-4-6`) via the Vercel AI Gateway and
asks for a 3–5 sentence analyst-style summary.

The narrative is **reporting only** — it never affects `overall_score` or
`decision`. This preserves the deterministic, defensible scoring surface that
compliance can trust while giving analysts a human-readable story.

Toggled via `ENABLE_NARRATIVE=1` so it can be cheaply disabled during batch
evaluation.

## API Surface

- `POST /analyze` — multipart PDF upload (+ optional `ocr_text` field), returns Report JSON
- `POST /analyze-batch` — accepts an array; returns array of reports (for corpus evaluation)
- `GET /healthz` — trivial liveness
- CLI: `tamper-detect path/to/file.pdf [--json | --pretty] [--no-narrative]`

## Test Corpus (10 Documents)

Generated by `testdata/generate.py`. Every document has a ground-truth entry in
`labels.json` naming the detector signals that should fire.

| # | Doc type | Tampering | Expected signals |
|---|---|---|---|
| 1 | Utility invoice | **Clean** — pdf-native via ReportLab, embedded fonts, single Producer | none (score < 0.15) |
| 2 | Utility invoice | Amount edited via pikepdf incremental update | `incremental_update_present`, `mod_date_after_create_date`, possibly `font_family_mix_in_line` |
| 3 | Utility invoice | Scanned + spliced amount (PIL paste at different JPEG quality, wrapped back in PDF) | `jpeg_ghost`, `ela_hotspots`, `noise_residual_inconsistency`, `text_layer_mismatches_render` |
| 4 | Sales tax permit | **Clean** | none |
| 5 | Sales tax permit | Name/business swapped via text-layer overlay (white rect + new text on top) | `text_over_image_overlay`, `text_layer_mismatches_render` |
| 6 | Sales tax permit | Re-saved through a Word-like path (Producer/Creator mismatch) | `producer_creator_mismatch`, possibly `font_not_embedded` |
| 7 | EIN letter | **Clean** — IRS-style layout, embedded monospace | none |
| 8 | EIN letter | EIN number replaced via pikepdf **incremental save** with a mixed font on the replacement digits | `font_family_mix_in_line`, `incremental_update_present` |
| 9 | EIN letter | Fully synthetic look-alike (rendered ReportLab doc noised + JPEG-recompressed to simulate diffusion artifacts, wrapped in PDF) | `ai_generated_image` (experimental), `template_mismatch`, `noise_residual_inconsistency` |
| 10 | Utility invoice | Copy-moved logo/stamp (duplicated logo covers underlying "PAID" watermark) | `copy_move`, `ela_hotspots` |

      "authentic": true,
      "expected_signals": [],
      "expected_decision": "pass"
    },
    {

**Note on realism:** these are synthetic and won't perfectly reflect real
production fraud. The corpus is for detector development — proving each
detector fires when it should. Follow-on work is a growing labeled real-world
set and re-tuning weights against it.

## Evaluation Harness

`tests/integration/test_corpus.py` iterates `labels.json`, runs `/analyze` on
each doc, and asserts:

1. Every `expected_signal` fires with `score ≥ 0.5`.
2. Clean docs get `decision == "pass"`.
3. Tampered docs get `decision != "pass"`.
4. Prints a confusion-matrix-style summary so weights can be tuned by
   inspection.

## Dependencies

Pinned in `pyproject.toml`:

- `pymupdf` — PDF parsing, page render, embedded image extraction, font info
- `pikepdf` — low-level PDF object model; used for detection *and* to build tampered test docs
- `reportlab` — clean test PDF generation
- `pillow`, `numpy`, `opencv-python` — pixel forensics
- `pytesseract` (+ system `tesseract`) — OCR-vs-render diff
- `scikit-image` — block matching for copy-move, SSIM
- `pyyaml` — weights config
- `fastapi`, `uvicorn` — service
- `typer` — CLI
- `pydantic` v2 — Report/Finding models
- `pytest`, `pytest-asyncio` — tests
- `anthropic` SDK via Vercel AI Gateway — narrative
- Optional extras `[ai-detect]`: `torch` + a small pretrained AI-image detector checkpoint. Kept optional so the base install stays lean.

## Configuration

- `weights.yaml` — signal weights + decision thresholds
- `.env`:
  - `ENABLE_NARRATIVE` (default `0`)
  - `AI_GATEWAY_API_KEY`
  - `NARRATIVE_MODEL` (default `anthropic/claude-sonnet-4-6`)
  - `TESSERACT_CMD` (optional override)
- Detector-level tunables live as named constants at the top of each detector module — no scattered magic numbers.
